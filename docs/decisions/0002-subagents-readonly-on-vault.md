# ADR-0002: Субагенты read-only на vault

- **Status:** accepted
- **Date:** 2026-04-11 *(ratified; решение в v2.0.0-alpha.1)*
- **Scope:** `agents/vault-worker.md` и все скиллы, которые его диспатчат (audit, dedup, cluster, process-inbox)

## Context

В v2.0 скиллы audit, dedup, cluster, process-inbox диспатчат субагентов (`vault-worker`) для параллельного сканирования больших vault-ов. Субагенты запускаются изолированно от главного контекста и видят только task prompt — без истории разговора, без пользовательского intent, без возможности переспросить.

Если субагент имеет право писать в vault, любой баг в нём или неверная интерпретация промпта приводит к массовой порче данных:
- Переместить 200 файлов в не ту папку
- Переписать frontmatter с неправильными значениями
- Удалить заметки, приняв их за дубликаты

Откатить такие изменения без git сложно, особенно если субагент успел обновить wikilinks по всему vault. А `obsidian_vault` под git — редкое исключение, не норма.

## Decision

**Субагенты `vault-worker` строго read-only на vault.**

Разрешено:
- `Read`, `Grep`, `Glob` на файлы vault
- `Bash` с read-only командами Obsidian CLI: `files`, `read`, `properties`, `tags`, `tag`, `search`, `search:context`, `links`, `backlinks`, `orphans`, `unresolved`, `eval`
- **Write только в `_temp/`** (для JSON-отчётов)

Запрещено (жёсткий constraint в system-prompt агента):
- Любые write-команды Obsidian CLI: `move`, `property:set`, `property:remove`, `append`, `prepend`, `create`, `delete`, `tags:rename`, `daily:append`
- `Edit` / `Write` на файлы vault (кроме `_temp/`)

**Return contract:** субагент возвращает главному контексту только summary 1–5 предложений, никаких сырых данных или списков файлов — это идёт в JSON-отчёт в `_temp/`.

## Alternatives considered

1. **Субагенты могут писать под trust level `cautious`/`balanced`, но не `auto`.** Отвергнуто: субагент запускается параллельно без полного контекста, даже в `cautious` режиме сложно гарантировать, что промежуточное подтверждение дошло корректно. Плюс архитектурно — если разделение есть, лучше сделать его абсолютным.
2. **Один главный Claude делает всё (scan + apply).** Отвергнуто: не масштабируется на 2000+ заметок. Главный контекст переполняется сырыми данными до того, как план сформирован, — и скилл падает на merge-фазе из-за context overflow.
3. **Отдельная роль `write-agent` с жёсткими constraint-ами на white-list операций.** Отвергнуто преждевременно (YAGNI): текущая архитектура «scan параллельно → apply в главном контексте» покрывает все use-cases без write-субагентов. Если apply-фаза станет bottleneck, можно вернуться к этой опции отдельным ADR.

## Consequences

**Плюсы:**
- Невозможен массовый write-accident от субагента — максимум они засорят `_temp/` мусорными JSON-ами
- Чёткое разделение ответственности: субагенты анализируют, главный контекст применяет
- Главный контекст видит только summary + JSON из `_temp/` → не переполняется сырыми данными
- Контракт агента крайне прост: «пишешь только в `_temp/`, возвращаешь 1–5 предложений» — сложно нарушить случайно

**Минусы:**
- Apply-фаза вся в главном контексте — не параллелится, даже когда операции независимы
- Нельзя выполнять «full auto» без участия главного Claude — на очень больших объёмах это теоретический bottleneck
- При сотнях apply-операций главный контекст всё равно может приблизиться к пределу (но это отодвинутая проблема — JSON-отчёты сильно компактнее сырого grep output)
