# ADR-0006: Русский UX, английские теги/код/коммиты

- **Status:** accepted
- **Date:** 2026-04-11 *(ratified; enforced в v1.2.0, 2026-03-29)*
- **Scope:** все скиллы, frontmatter spec, coding conventions

## Context

Плагин изначально разрабатывался одним русскоязычным автором для личного vault-а, который смешивает русский и английский контент. По мере использования столкнулись с конкретными проблемами:

- **Русские теги** (`#ml/метрики`) создавали межъязыковые дубликаты (`#ml/metrics` vs `#ml/метрики`) — `/audit` засорялся одинаковыми по смыслу, но разными по написанию тегами
- **Русские имена файлов** нормальны для контентных заметок, но вызывают проблемы с английскими аббревиатурами, которые нельзя переводить: Git, ML, SemVer, RMSLE, YAML, API
- **Коммит-сообщения** нужно оставлять на английском для совместимости с инструментарием: changelog generators, conventional commits, GitHub PR templates
- **UX-сообщения** скиллов должны быть на русском — иначе плагин для русскоязычного автора становится неудобным («ну и зачем я его тогда пишу»)

Нужен чёткий набор правил, разделяющий языки по слоям.

## Decision

Жёсткое разделение языков по слоям:

| Слой | Язык | Обоснование |
|---|---|---|
| **UX / сообщения пользователю** (в промптах скиллов) | Русский | Целевая аудитория — русскоязычный автор |
| **Имена файлов заметок** | Русский **по умолчанию**, английский для непереводимых терминов (Git, ML, SemVer, RMSLE, API) | Контент в основном русский |
| **Теги (YAML frontmatter + inline)** | **Английский, строго иерархические** (`ml/metrics`, `dev/python`, `finance/bonds`) | Избегаем cross-language duplicates, унификация поиска |
| **Код** (Python/JS/Bash) | Английский | Индустриальный стандарт |
| **Комментарии в коде** | Английский | Консистентность с кодом |
| **Коммит-сообщения** | Английский | Conventional commits, changelog generators, совместимость с tooling |
| **Документация README** | Двуязычная (`README.md` EN + `README.ru.md` RU) | Плагин может быть полезен не только автору |
| **ADR, known-issues, specs** (этот реестр) | Русский (авторские рассуждения) с английскими терминами для кода/инструментов | Это dev-документация для русскоязычного автора |

**Enforcement:**
- `/audit` детектит non-English теги в frontmatter и предлагает исправить (v1.2.0)
- `/rename-tags` корректирует casing и cross-language дубли
- Все скиллы используют русскую речь в интерактивных промптах
- `references/frontmatter-spec.md` явно требует `tags: required (English)`

## Alternatives considered

1. **Всё на английском.** Отвергнуто: UX для русскоязычного автора неудобен, имена файлов теряют смысл. Например, «Механизм внимания в трансформерах» ≠ «Attention mechanism in transformers» по гуглимости внутри русскоязычного vault-а — автор ищет по русским ключевым словам, которые он реально использует в мышлении.
2. **Всё на русском.** Отвергнуто: теги ломают совместимость (cross-language dupes при добавлении любой English-заметки), коммиты выпадают из стандартов (`feat: добавить ...` не парсится changelog generators), код на русском — плохая идея в общем случае (import `пандас` as pd).
3. **Автоопределение языка заметки и применение той же локали к тегам.** Отвергнуто как сложное и хрупкое: как определить язык смешанной заметки? Что делать с одним тегом, применённым к заметкам на разных языках? Жёсткое правило «теги всегда английские» проще в реализации и проверке, а для русских заметок — это просто перевод keywords, что даже полезно для улучшения английского поиска.
4. **Английские теги, но русские для специфичных доменов (философия, литература).** Отвергнуто: как определить «специфичный домен»? Правило с исключениями — это не правило. Если автор решит, что ему нужен `#философия/экзистенциализм`, он может добавить его вручную — audit будет ругаться, но это осознанный выбор.

## Consequences

**Плюсы:**
- **Нет cross-language tag duplicates** → audit чище, кластеризация по тегам работает предсказуемо
- **Коммиты и код совместимы** с экосистемой (CI, changelog generators, conventional commits)
- **UX остаётся комфортным** для целевой аудитории — на русском там, где это имеет значение
- **Имена файлов отражают авторское мышление** — поиск по vault находит то, что автор реально пишет

**Минусы:**
- **Пользователь, переводящий tag на английский**, должен знать оба языка — лёгкий барьер для неанглоговорящих, но целевая аудитория этим не ограничена (автор билингвален)
- **Правило «русский для файлов, английский для аббревиатур»** субъективно: требует judgment для пограничных случаев. Например, «Kubernetes» оставить как есть или написать «Кубернетес»? «Flux» → оставить? «PyTorch» → явно не переводится. Разрешается **авторским выбором**, без жёсткого правила.
- Документация в репо частично двуязычная (README — оба языка, ADR — русский) — может вызвать вопросы у новых пользователей, почему одни файлы на одном языке, другие на другом.
