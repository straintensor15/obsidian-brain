# ADR-0001: Trust levels — три уровня (cautious/balanced/auto)

- **Status:** accepted
- **Date:** 2026-04-11 *(ratified; фактически принято в v2.0.0-alpha.1, 2026-03-31)*
- **Scope:** все 7 скиллов плагина

## Context

Плагин ориентирован на vault-ы 2000+ заметок. Любая массовая операция (применить 87 фиксов frontmatter, переместить 12 файлов, смержить 5 дубликатов) стоит дорого, если её откатывать, и ещё дороже, если она ломает wikilinks по всему vault.

Одновременно:
- **Новый пользователь** хочет видеть, что именно делает скилл, и готов подтверждать каждое действие.
- **Опытный пользователь** на повторных запусках хочет скорости — «применить всё, показать отчёт».
- Некоторые операции (`obsidian move`, dedup merge) **необратимы** и требуют подтверждения независимо от предпочтений пользователя.

Нужна единая модель granularity подтверждений для всех 7 скиллов, чтобы пользователь не учил новый UX для каждого.

## Decision

Ввести три уровня подтверждения (trust levels), применяемых универсально ко всем скиллам:

| Level | Behavior | Default for |
|---|---|---|
| `cautious` | Подтверждение каждого элемента | dedup |
| `balanced` | Подтверждение по категориям/группам | audit, cluster, process-inbox, rename-tags |
| `auto` | Применить всё, показать итоговый отчёт | — (только по явному выбору пользователя) |

**Жёсткие ограничения** (override пользовательский выбор, работают даже в `auto`):
- `obsidian move` — минимум `balanced` (vault-wide wikilinks side effects)
- dedup merge — минимум `balanced` (необратимое слияние контента)
- Первый запуск любого скилла — **не `auto`** (пользователь должен хотя бы раз увидеть, что делает скилл)

## Alternatives considered

1. **Бинарный режим (`dry-run` vs `apply-all`).** Простейшая модель: показать список изменений → пользователь либо говорит «примени всё», либо копирует ID руками. Отвергнуто: в dedup/improve это деградирует до ручной работы — на 30 парах дубликатов пользователь не хочет ни просматривать всё без возможности вмешаться, ни собирать ID обратно в CLI. Категоризация по группам (`balanced`) — это именно тот промежуточный режим, ради которого схема вообще существует.

2. **Пять уровней (`paranoid` / `cautious` / `balanced` / `trust` / `auto`).** Больше контроля, но cognitive load растёт быстрее, чем практическая польза: граница «paranoid vs cautious» или «trust vs auto» требует объяснения, которое не умещается в одно предложение. Правило «если нельзя в одном предложении — нельзя в UX» отсекает лишнее: три уровня — максимум, где каждый имеет чёткое поведенческое отличие (per-item / per-group / all-at-once).

3. **Per-skill hardcoded policy без выбора пользователя.** Каждый скилл просто знает свой «правильный» режим: dedup всегда спрашивает, audit всегда группирует. Отвергнуто: убивает сценарий повторного запуска — пользователь, который уже прогнал `/audit` и увидел, что скилл делает, во втором запуске хочет `auto`, а не тот же батч вопросов. Дефолт по скиллу остался (см. таблицу в Decision), но как **стартовая точка**, а не как жёсткая политика.

4. **Trust level на уровне категории issue** (например, «frontmatter-фиксы auto, move-операции cautious» внутри одного скилла). Привлекательно в теории: tunable гранулярность там, где она нужна. Отвергнуто как over-engineering: пользователь платит сложностью настройки за выигрыш, который уже достигается проще — **жёсткими ограничениями** (`obsidian move` → минимум `balanced` независимо от выбора). Hard constraints дают тот же безопасностный эффект без matrix-конфига «N скиллов × M категорий».

## Consequences

**Плюсы:**
- Единая модель взаимодействия во всех скиллах — пользователь не учит новый UX для каждого
- Хард-ограничения защищают от необратимых ошибок даже в `auto` режиме
- Default `balanced` на большинстве скиллов — разумный середняк между безопасностью и скоростью
- Простая mental model: три слова, каждое означает понятное

**Минусы:**
- Три уровня — ещё один вопрос в начале каждого скилла. На коротких запусках это трение.
- `auto` с ограничениями — это не «настоящий auto»: пользователь всё равно видит вопросы на move/merge. Нужно объяснять в UX, иначе воспринимается как баг.
- Промежуточное состояние: в `cautious` у пользователя есть опция «применить все оставшиеся в категории» — это фактически мягкий upgrade до `balanced` на лету. Гибко, но усложняет описание поведения.
