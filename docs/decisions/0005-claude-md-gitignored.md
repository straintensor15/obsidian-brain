# ADR-0005: `CLAUDE.md` — gitignored локальная шпаргалка

- **Status:** accepted
- **Date:** 2026-04-11 *(ratified; решение в commit `b858bc3`, 2026-03-31)*
- **Scope:** `CLAUDE.md` в корне плагина

## Context

Изначально `CLAUDE.md` плагина трекался в git как документация разработки. По ходу развития v2.0 стало понятно, что он играет роль **личного боевого блокнота для Claude**, а не публичного API плагина:

- Содержит навигацию к локальным (gitignored) `docs/specs/`, `docs/plans/`
- Помечает local-only пути и конвенции автора
- Упоминает устаревшие артефакты разработки и in-progress ветки, которые не стоит показывать пользователям плагина из marketplace
- Публичная документация для пользователей — в `README.md` / `README.ru.md` + `.claude-plugin/plugin.json`

Коммит в git делал `CLAUDE.md` частью дистрибутива плагина. Пользователи, установившие `obsidian-brain` через marketplace, получали бы этот файл в `~/.claude/plugins/cache/.../obsidian-brain/CLAUDE.md`, где он либо подхватывался бы как рабочий CLAUDE.md (если они открывали плагинный репо как working dir), либо просто мешался бы в дереве файлов.

## Decision

**`CLAUDE.md` добавлен в `.gitignore`** и удалён из трекинга (commit `b858bc3 chore: add CLAUDE.md to .gitignore and remove from tracking`).

Разделение ответственности:
- **`CLAUDE.md`** (локальный, gitignored) — для Claude. Навигация, конвенции разработки, личные заметки автора.
- **`README.md` / `README.ru.md`** (в git) — для людей. Установка, использование, примеры.
- **`.claude-plugin/plugin.json`** (в git) — для плагин-системы Claude Code. Метаданные, версия.
- **Родительский `my_plugins/CLAUDE.md`** (локальный) — общие правила разработки всех плагинов автора, наследуется Claude Code при работе в `obsidian-brain/`.

Синхронизация между устройствами автора (yasdr ↔ BestPC) — через `~/.claude` репо автора, не через репо плагина.

## Alternatives considered

1. **Оставить в git, использовать условные секции с комментариями.** Отвергнуто: нет надёжного механизма «этот блок только для dev, а этот для пользователей». Смешивать dev-контекст и пользовательскую документацию в одном файле — рецепт утечек личной информации в публичный дистрибутив.
2. **Переименовать в `CLAUDE.dev.md` и оставить в git.** Отвергнуто: Claude Code по конвенции ищет именно `CLAUDE.md`. Переименование ломает автоматическую подгрузку — файл становится обычным md, который Claude не читает без явной инструкции.
3. **Вынести содержимое в `docs/dev-notes.md`.** Отвергнуто: теряется автоматическая загрузка в контекст. `CLAUDE.md` загружается всегда при старте сессии; любой другой файл — только по явной инструкции или вопросу пользователя.
4. **Использовать local override через `.claude.local.md`** (gitignored by convention). Отвергнуто: нестандартное имя в экосистеме, не все tools его понимают. Проще явно положить `CLAUDE.md` в `.gitignore`.

## Consequences

**Плюсы:**
- **Пользователи плагина не получают чужой dev-контекст** при установке через marketplace
- **Автор может держать в `CLAUDE.md` личные заметки** без цензуры — нет риска случайно запушить
- **Чёткое разделение:** `CLAUDE.md` = для Claude (локально), `README*` + `plugin.json` = для пользователей
- Нет необходимости поддерживать двуязычный `CLAUDE.md` (русский для автора, английский для пользователей)

**Минусы:**
- **На чистом клоне репо `CLAUDE.md` отсутствует** — Claude без него работает хуже (теряет навигацию, конвенции, known gotchas). Нужно либо воссоздавать вручную, либо синхронизировать через `~/.claude` репо автора.
- **Неочевидно для человека,** впервые открывшего репо плагина: он видит в `README` ссылки на разные места, но не понимает, где искать «карту проекта». Частично смягчено тем, что этот ADR и `docs/_INDEX.md` коммитятся.
- При разработке плагина на новом устройстве нужен ручной шаг: создать или скопировать `CLAUDE.md`.
