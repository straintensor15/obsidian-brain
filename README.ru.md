# obsidian-brain

Плагин Claude Code для управления Obsidian-хранилищем через CLI. Набор скиллов для обработки, организации и поддержки базы знаний, не выходя из терминала.

## Требования

- Obsidian >= 1.12.4 с включённым CLI
- **Установщик Obsidian должен быть актуальным** — если вы видите предупреждение `Your Obsidian installer is out of date`, скачайте последний установщик с [obsidian.md/download](https://obsidian.md/download) и переустановите (хранилище и настройки сохранятся). Устаревший установщик может привести к тихим сбоям CLI-команд записи.
- Obsidian должен быть запущен при использовании скиллов
- Claude Code

### Включение Obsidian CLI

1. Obsidian → Настройки → Основные → Интерфейс командной строки
2. Включите **Register CLI**
3. Проверьте: `obsidian version` в терминале

### Windows: обёртка для CLI (важно)

На Windows `obsidian` может резолвиться в `Obsidian.exe` (GUI-версия) вместо `Obsidian.com` (консольная версия). GUI exe ломается на подкомандах с двоеточием и параметрами (`property:set`, `daily:append` и т.д.) с кодом выхода 127.

**Решение:** создайте скрипт-обёртку `~/bin/obsidian` (убедитесь, что `~/bin` есть в PATH):

```bash
mkdir -p ~/bin
cat > ~/bin/obsidian << 'EOF'
#!/usr/bin/env bash
exec '/c/Users/<username>/AppData/Local/Programs/Obsidian/Obsidian.com' "$@"
EOF
chmod +x ~/bin/obsidian
```

Скрипт-обёртка работает как в интерактивных, так и в неинтерактивных shell (в отличие от alias).

## Скиллы

### `/process-inbox` — Обработка входящих заметок

Обрабатывает сырые заметки из `_inbox/`: определяет тему, задаёт frontmatter и теги, находит связанные заметки, добавляет wikilinks, перемещает в целевую папку с автоматическим обновлением ссылок.

**Типичный сценарий:**
1. Закидываете заметки в `_inbox/` (после встречи, лекции или чтения)
2. Запускаете `/process-inbox`
3. Для каждой заметки: смотрите предложенное имя, теги, папку и найденные связи
4. Подтверждаете или корректируете план
5. Заметки организованы, связаны wikilinks и проиндексированы Obsidian

---

### `/decompose` — Декомпозиция монолитных заметок

Разбивает большие заметки (>10 КБ) на атомарные и создаёт Map of Content (MOC).

**Типичный сценарий:**
1. Конспект книги на 5000 слов лежит в `literature/`
2. Запускаете `/decompose` (или плагин сам предложит кандидатов)
3. Смотрите план: какие разделы станут отдельными заметками, какие останутся в MOC
4. Подтверждаете
5. Результат: 8 атомарных заметок в `notes/` + MOC в `_moc/` со ссылками

---

### `/dedup` — Поиск и объединение дубликатов

Трёхступенчатый поиск дубликатов (по именам → тегам → содержимому) с интерактивным объединением.

**Типичный сценарий:**
1. Запускаете `/dedup`
2. Плагин находит пары с разной уверенностью (high / medium / low)
3. Для каждой пары: смотрите оба файла, выбираете основной
4. Дубликат архивируется в `_archive/merged/`, все ссылки обновляются автоматически

---

### `/cluster` — Кластеризация и анализ графа

Два режима:

**`analyze` — диагностика графа:**
1. Запускаете `/cluster` → выбираете `analyze`
2. Получаете отчёт: хабы (самые связанные заметки), сироты, тупики, битые ссылки, слабые кластеры
3. Для каждой проблемы — рекомендация (создать связи, добавить в MOC, архивировать)

**`group` — группировка в MOC/summary:**
1. Запускаете `/cluster` → выбираете `group`
2. Плагин находит кластеры связанных заметок (по тегам, ссылкам, содержимому)
3. Для каждого кластера: предлагает MOC (разные аспекты) или summary-заметку (общий тезис)
4. Подтверждаете состав и тип → MOC в `_moc/` или summary в `notes/`

---

### `/audit` — Аудит качества хранилища

Два режима:

**`quick` — healthcheck за 30 секунд:**
1. Запускаете `/audit` → выбираете `quick`
2. Три быстрые проверки: сироты, битые ссылки, одиночные теги
3. Краткий отчёт с цифрами

**`full` — полный аудит с исправлениями:**
1. Запускаете `/audit` → выбираете `full`
2. Проверки по 5 категориям: frontmatter, теги, ссылки, размещение, содержимое
3. Сводный отчёт с количеством проблем по категориям
4. Выбираете: исправить всё / только категорию X / пропустить
5. Интерактивный режим: для каждой проблемы — предложение исправления → подтверждение

---

### `/rename-tags` — Рефакторинг таксономии тегов

Интерактивный рефакторинг таксономии тегов хранилища.

**Типичный сценарий:**
1. Запускаете `/rename-tags`
2. Видите полное дерево тегов с количеством заметок
3. Плагин определяет: одиночные теги, дубликаты с разным регистром, плоские теги, которые можно сделать иерархическими
4. Утверждаете план переименования (можно отредактировать)
5. Теги переименованы по всему хранилищу одной командой

---

### `/improve` — Улучшение содержимого заметки

Интерактивное улучшение содержимого: структура, форматирование, таблицы, перефразирование, резюме, wikilinks.

**Типичный сценарий:**
1. Запускаете `/improve` и указываете заметку
2. Видите меню операций с оценкой релевантности (high / medium / low)
3. Выбираете нужные операции
4. Для каждой операции: превью «до → после» → подтверждение или пропуск
5. Итоговый отчёт о применённых изменениях

**Доступные операции:**

| # | Операция | Что делает |
|---|----------|------------|
| 1 | Структура | Заголовки, реорганизация разделов, пустые секции |
| 2 | Формат | Выделение терминов, списки, callouts |
| 3 | Таблицы | Перечисления → markdown-таблицы |
| 4 | Перефразирование | Лучшие формулировки, убрать воду |
| 5 | Резюме | TL;DR с ключевыми тезисами |
| 6 | Wikify | Добавить `[[wikilinks]]` на существующие заметки |

## Настройка

Перед первым использованием настройте файлы в `references/` под ваше хранилище:

```bash
# Скопируйте шаблон конфигурации и отредактируйте
cp references/vault-config.example.md references/vault-config.md
# Откройте references/vault-config.md и укажите путь к хранилищу и структуру папок
```

Справочные файлы:

- `references/vault-config.md` — **путь к хранилищу и структура папок** (в gitignore, копируйте из `.example.md`)
- `references/frontmatter-spec.md` — шаблоны frontmatter и правила полей
- `references/cli-operations.md` — справочник команд Obsidian CLI
- `references/interaction-patterns.md` — паттерны взаимодействия (язык, подтверждения, отчёты)

## Ожидаемая структура хранилища

```
_inbox/        ← Входящие заметки (обрабатываются /process-inbox)
notes/         ← Атомарные заметки
literature/    ← Конспекты книг/курсов/статей
projects/      ← Проектные заметки
_moc/          ← Maps of Content (навигационные файлы)
daily/         ← Ежедневные заметки
_archive/      ← Архив (неактивные + объединённые дубликаты)
_templates/    ← Шаблоны Obsidian
_assets/       ← Вложения (изображения, PDF)
```

## Безопасность

- Файлы никогда не удаляются — только архивируются
- Перемещение и переименование только после подтверждения пользователя
- Wikilinks обновляются автоматически через CLI при перемещении файлов
- Всё содержимое сохраняется — ничего не теряется
- Git-коммиты после каждой операции с указанием конкретных файлов
